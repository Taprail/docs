Every order you create comes with two handles:

```json
{
  "deep_link": "https://pay.taprail.app/pay?o=<order_id>",
  "qr_payload": "https://pay.taprail.app/pay?o=<order_id>"
}
```

How you present them depends on where your customer is.

## Web (recommended): the JS SDK

`Taprail.open(intent)` from [`@taprail/checkout`](/sdk/js) does the right thing per device:

- **Desktop**: opens a 420×620 modal with the order summary and a QR; the customer scans
  with their phone and approves in the Taprail app. The SDK polls the
  [public order endpoint](/api-reference/public-order) every 2.5 s and resolves when the
  order reaches a terminal status.
- **Mobile**: opens the Taprail app directly and shows a small waiting overlay on your page.
  The customer approves in the app and switches back; the SDK polls the
  [public order endpoint](/api-reference/public-order) and resolves on a terminal status,
  the same as desktop. Your webhook is still the authoritative result.

## Direct deep link

Skip the SDK entirely: put `deep_link` behind your own "Pay with Taprail" button, or render
`qr_payload` as a QR at a physical counter. Then either poll
[`GET /v1/merchant/orders/:id/status`](/api-reference/get-order-status) from your server or
wait for the webhook.

## Browser return URL

Set a **return URL** in the dashboard (**Payments → Webhooks → Browser return URL**) and the
hosted checkout sends the customer's browser back to your site once the order reaches a
terminal status, with the order id and status appended as query params:

```text
https://yourstore.com/checkout/complete?order=<order_id>&status=approved
```

- Must be `https://`. Leave it blank to keep customers on the Taprail confirmation screen.
- It's a UX convenience. **Never fulfil from query params alone**; confirm against the
  webhook or `GET /v1/merchant/orders/:id`.

## Order lifecycle

```text
created ──▶ awaiting_user ──┬──▶ approved ──▶ refunding ──▶ refunded
   │             │          │
   │             │          ├──▶ payment_unconfirmed ──▶ approved / declined
   │             │          │
   │             │          └──▶ declined
   │             │
   └─────┬───────┴──▶ expired / cancelled
         ▼
   (terminal: approved · declined · expired · cancelled · refunded)
```

| Status | Terminal | Meaning |
|---|---|---|
| `created` | | Order exists; the customer hasn't opened it yet |
| `awaiting_user` | | Opened in the app, not yet authorized |
| `payment_unconfirmed` | | The charge happened but its confirmation was lost in transit. Taprail re-verifies and moves it to `approved` or `declined`. **Don't treat this as a failure** — a real charge is never auto-declined |
| `approved` | ✓ | Authorized and charged. Fulfil on this, via the webhook |
| `refunding` | | A refund is in flight. Settles to `refunded` |
| `refunded` | ✓ | You refunded an approved order |
| `declined` | ✓ | Charge failed or the customer declined (`decline_reason` says why) |
| `expired` | ✓ | Not completed within the 30-minute window |
| `cancelled` | ✓ | You cancelled it before payment |

`payment_unconfirmed` and `refunding` are transient: they resolve to a terminal status on
their own, usually within a minute. Build your client to wait them out rather than surfacing
them to the customer as an outcome.

<Note>
  Orders expire **30 minutes** after creation if unpaid. Expiry produces an `order.expired`
  webhook, so abandoned carts clean themselves up.
</Note>
