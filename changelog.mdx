<Update label="June 2026" description="Themeable checkout (SDK 0.2.2)">

**JS SDK 0.2.2 тАФ themeable, refreshed checkout.** The modal got a visual refresh тАФ it animates
in, shows a Taprail wordmark and a loader while the embed comes up, and the mobile waiting card
now shows the amount + merchant and resolves the moment the shopper returns from the app. It's
also **themeable**: pass a `theme` to `init()` (a default for every checkout) or to `open()`
per call тАФ `mode` (`auto` default / `light` / `dark`), `accent`, `surface`, `text`, `muted`,
`backdrop`, `radius`, and `fontFamily`. Defaults stay monochrome; `auto` follows the OS, and the
modal respects `prefers-reduced-motion`. See the [JS SDK theming](/sdk/js#theming) section.

</Update>

<Update label="June 2026" description="Mobile checkout resolves (SDK 0.2.1), payment-recovery states">

**JS SDK 0.2.1 тАФ mobile checkout resolves.** `open()` on mobile used to send the customer
to an intermediate web page, and the promise never settled there. It now opens the Taprail
app directly from the tap gesture and shows a small waiting overlay on your page while it
polls. The promise resolves with the terminal result on mobile just like on desktop, so you
can route to a receipt after a mobile approval. The webhook stays the source of truth.

**Payment-recovery states are now observable.** Two non-terminal statuses can appear on
order reads. `payment_unconfirmed` parks a charge whose confirmation was lost in transit;
Taprail re-verifies it and either promotes it to `approved` or `declined` тАФ a real charge is
never auto-declined. `refunding` marks a refund that's in flight. Don't treat either as a
failure; wait for a terminal status. See [the order lifecycle](/checkout-flows#order-lifecycle).

**Asynchronous refunds.** [Refund](/api-reference/refund-order) can return
`{ "status": "processing" }` when the processor call is deferred. The refund completes on its
own and the `order.refunded` webhook fires when it does. Refunds stay idempotent: a second
refund of the same order is a no-op.

</Update>

<Update label="June 2026" description="Rate limits, granular errors, return URL, SDK 0.2.0">

**JS SDK 0.2.0.** `CheckoutResult` gained `return_url`, and `open()` gained an opt-in
`redirect` option that sends the browser to your configured return URL after a terminal
status. The modal now verifies the origin of embed messages, locks background scroll,
and manages focus properly. Transient polling errors no longer spam `onError`.

**Rate limiting on the merchant API.** Each merchant is limited to **300 requests per
minute** across all `/v1/merchant/*` endpoints. Exceeding it returns
`429 too_many_requests`. See [rate limits](/rate-limits).

**Granular error codes.** Expected failures now return specific, stable codes instead of
status-level ones: `invalid_api_key`, `live_mode_disabled`, `duplicate_reference`,
`amount_invalid`, `order_not_refundable`, and friends. Validation errors gained a `param`
field naming the offending field. Coarse codes (`bad_request`, `conflict`) still appear on
generic failures, so existing clients keep working. The maintenance pause now correctly
returns `503 unavailable_maintenance` (previously `400`).

**Browser return URL.** Merchants can configure a return URL in the dashboard; the hosted
checkout sends the customer back to it with `order` and `status` query params once the
order goes terminal. The public order view gained a `return_url` field.

</Update>
