```bash
npm install @taprail/checkout
```

Zero runtime dependencies, ESM/CJS/UMD builds, optional React bindings (React ≥ 18), and a
[themeable](#theming) modal.

## Open a checkout

```javascript
import { open } from "@taprail/checkout";

// `intent` is the JSON your server got from POST /v1/merchant/orders.
const result = await open(intent, {
  onSuccess: (r) => markPaidInUi(r.order_id),
  onError: (e) => showError(e),
  onClose: () => trackAbandon(),
});

// result:
// { status: "approved" | "declined" | "expired" | "cancelled" | "refunded" | "closed",
//   order_id: "...",
//   transaction_reference: "tprm_..." | null,
//   return_url: "https://yourstore.com/..." | null }
```

`open` accepts the **intent object**, a bare **order id**, or a **deep link URL**, whichever
is handy.

If you've configured a [browser return URL](/checkout-flows#browser-return-url) and would
rather not handle the result yourself, pass `redirect: true` and the SDK navigates there
after a terminal status, appending `?order=<id>&status=<status>`:

```javascript
await open(intent, { redirect: true });
```

What happens per device:

- **Desktop** → modal (iframe) with the order summary + QR. The SDK polls the public order
  endpoint every 2.5 s and resolves on a terminal status, or `{ status: "closed" }` if the
  customer dismisses it.
- **Mobile** → opens the Taprail app directly (no intermediate page) and shows a small
  waiting overlay on your page. The customer approves in the app, switches back, and the
  promise resolves with the terminal result — the same shape you get on desktop. The overlay
  has an "Open Taprail" button as a fallback if the app didn't come forward.

<Note>
  `open()` must be called inside the customer's tap or click. On mobile the SDK launches the
  app from that gesture; browsers ignore app launches that aren't user-initiated.
</Note>

<Warning>
  The SDK result is for UX (spinners, confirmation screens). Fulfil orders from the signed
  [webhook](/webhooks/overview), or by confirming with `GET /v1/merchant/orders/:id` from
  your server. On mobile the customer can always close the app without returning, so treat a
  resolved promise as a hint, never as settlement.
</Warning>

## Theming

The modal chrome — backdrop, panel, the mobile waiting card, and buttons — follows a theme.
Set a default for every checkout with `init`, or pass one per call to `open`:

```javascript
import { init, open } from "@taprail/checkout";

// App-wide default (mode defaults to "auto" — follows the OS)
init({ theme: { accent: "#2f6df6" } }); // "auto" (default) | "light" | "dark"

// Or per checkout, with your brand
await open(intent, {
  theme: {
    mode: "light",
    accent: "#2f6df6",   // the "Open Taprail" button + focus rings
    accentText: "#ffffff",
    radius: 16,          // modal corner radius, px
  },
});
```

Every field is optional and falls back to the monochrome default for the resolved `mode`:

| Field | Colors |
|---|---|
| `mode` | `auto` (default — follows the OS), `light`, or `dark` |
| `accent` / `accentText` | The primary button and focus rings |
| `surface` | The modal / card background |
| `text` / `muted` | Primary and secondary text |
| `backdrop` | The scrim behind the modal (include your own alpha) |
| `radius` | Modal corner radius, in px |
| `fontFamily` | Font for the chrome |

The modal animates in, respects `prefers-reduced-motion`, and traps focus. Give colors as hex.
In React, pass `theme` to `<TaprailButton theme={…} />` or as an option to `useTaprailCheckout().open`.

## One-shot status check

```javascript
import { getStatus } from "@taprail/checkout";

const { status, transaction_reference } = await getStatus(intent);
```

## React

```tsx
import { TaprailButton, useTaprailCheckout } from "@taprail/checkout/react";

// Drop-in button
<TaprailButton
  intent={intent}
  onSuccess={(r) => router.push(`/orders/${r.order_id}/thanks`)}
>
  Pay ₦7,500
</TaprailButton>

// Or the hook for custom UI
const { open, pending } = useTaprailCheckout();
<button disabled={pending} onClick={() => open(intent)}>
  {pending ? "Processing…" : "Pay"}
</button>
```

## Script tag (no bundler)

```html
<script src="https://unpkg.com/@taprail/checkout"></script>
<script>
  Taprail.open(intent, { onSuccess: handlePaid });
</script>
```

## Pointing at staging

```javascript
import { init } from "@taprail/checkout";

init({
  apiBase: "https://api.staging.taprail.app",
  embedBase: "https://pay.staging.taprail.app",
});
```

Defaults are the production origins, so most integrations never call `init`.
